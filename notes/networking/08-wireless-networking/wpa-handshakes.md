# WPA Handshakes

> The WPA Handshake is the cryptographic conversation between a Wi-Fi client and an Access Point that proves both parties know the password and securely generates the encryption keys for the session.

## Overview
When you type a password into a website, it is sent over the network to be verified. In Wi-Fi, sending the password over the air—even encrypted—is incredibly dangerous. 

Instead of sending the password, the Access Point and the client engage in the **4-Way Handshake** (in WPA2). They use complex math to prove to each other that they *both* possess the correct password, without ever actually transmitting it. During this process, they construct unique, temporary encryption keys specifically for that user's session.

## Why It Matters
Understanding the mechanics of the handshake is the absolute core of wireless penetration testing. The standard WPA2 4-Way Handshake is the primary vulnerability targeted by hackers. If a cybersecurity professional understands exactly what data is exchanged in the four steps, they understand how to capture it, how to crack it offline, and why WPA3 was invented to replace it.

## Core Concepts
Before the handshake begins, both devices possess the **PSK (Pre-Shared Key)**—the actual Wi-Fi password. They also know the SSID (network name). 

Using the PSK and SSID, both devices independently calculate the **PMK (Pairwise Master Key)**. The PMK is the foundation for the handshake.

The goal of the handshake is to generate the **PTK (Pairwise Transient Key)**. The PTK is the actual, temporary encryption key used to protect the user's web traffic.

To create the PTK, you need four ingredients:
1. The PMK (Derived from the password).
2. The Authenticator MAC Address (The Access Point's MAC).
3. The Supplicant MAC Address (The Client's MAC).
4. Two **Nonces** (Random, one-time-use numbers generated by the AP and the Client).

## How It Works (The WPA2 4-Way Handshake)
The client attempts to connect to the AP.
1. **Message 1 (AP -> Client):** The AP generates a random number (ANonce) and sends it in cleartext to the client.
   *The client now has all 4 ingredients. It quietly calculates the PTK.*
2. **Message 2 (Client -> AP):** The client generates its own random number (SNonce). It sends the SNonce to the AP in cleartext. Crucially, it attaches a **MIC (Message Integrity Code)** to this message, created using the newly calculated PTK.
   *The AP receives the SNonce. It now has all 4 ingredients. It calculates its own version of the PTK. It checks the MIC. If the math matches, the AP knows the client has the correct Wi-Fi password.*
3. **Message 3 (AP -> Client):** The AP sends a message saying "Math verified, let's install the keys." It also includes the GTK (Group Temporal Key), which is used to decrypt broadcast/multicast traffic on the network.
4. **Message 4 (Client -> AP):** The client replies with a final ACK. 
*The handshake is complete. All further traffic is heavily encrypted using AES and the PTK.*

## Components / Types
- **EAPOL (Extensible Authentication Protocol over LAN):** The specific type of frame used to carry the 4-way handshake messages. When capturing handshakes in Wireshark, you filter for EAPOL packets.
- **WPA3 SAE (Dragonfly Handshake):** The modern replacement for the 4-way handshake. It uses a "commit and confirm" exchange that is immune to the offline dictionary attacks that plague WPA2.

## Practical Examples
- **The Coffee Shop Attack:** An attacker sits in a cafe with `airodump-ng` running. They wait for a new customer to walk in and connect to the Wi-Fi. The attacker captures the 4 EAPOL packets flying through the air. The attacker goes home, loads the capture into Hashcat, and tells the computer: "Guess a password, mix it with the Nonces from the capture, and see if the resulting MIC matches the MIC in Message 2." If it matches, the password is cracked.

## Security Considerations
- **Offline Dictionary Attacks:** Because Message 1 and Message 2 are sent in cleartext, an attacker has the ANonce, SNonce, MAC addresses, and the MIC. They have the entire mathematical equation, missing only the password variable. They can run billions of guesses a second on high-end GPUs completely offline.
- **Deauthentication Attack:** Attackers don't want to wait hours for someone to naturally connect to the network to capture a handshake. They spoof a "Deauth" frame, forcing an already-connected user off the network. The user's device instantly, automatically reconnects, generating a brand-new 4-Way Handshake for the attacker to capture on demand.

## Commands / Configuration Examples
### Pentesting (Capturing the Handshake with Aircrack-ng)
```bash
# 1. Listen to the target network
sudo airodump-ng -c 6 --bssid 00:11:22:33:44:55 -w handshake_capture wlan0mon

# 2. In a second terminal, force a connected client to deauthenticate 
# to trigger a reconnect and capture the EAPOL handshake
sudo aireplay-ng -0 5 -a 00:11:22:33:44:55 -c 99:88:77:66:55:44 wlan0mon

# Watch the first terminal. When it says "WPA handshake: 00:11:22:33:44:55", you have the data.
```

### Cracking the Handshake
```bash
# Use a wordlist to attempt to crack the captured handshake offline
aircrack-ng -w /usr/share/wordlists/rockyou.txt handshake_capture.cap
```

## Troubleshooting
- **Handshake Fails (Wrong Password):** If a user types the wrong password, Message 2 will still be sent, but the MIC calculation will be completely wrong. The AP will check the MIC, realize it fails, drop the connection, and refuse to send Message 3.
- **Incomplete Captures:** Pentesters often capture Message 1 and Message 2, but the client moves out of range before completing Message 4. Advanced tools like `hashcat` can actually crack a password using only Message 1 and 2, but it is always preferred to capture the full 4-way exchange.

## Interview Questions
- Why does the WPA2 handshake use random "Nonces"? (Answer: To ensure the resulting session key (PTK) is completely unique for every single connection, preventing replay attacks).
- Does the actual Wi-Fi password get transmitted during the 4-way handshake?
- Explain the role of the MIC (Message Integrity Code) in Message 2 of the handshake.
- Why is it possible to crack a WPA2 password completely offline?

## Summary
The WPA2 4-Way Handshake is an elegant cryptographic dance that securely establishes encryption keys without transmitting secrets. However, its reliance on a verifiable hash (the MIC) derived from the password makes it susceptible to offline brute-force attacks, a flaw that fundamentally shaped wireless security practices for nearly two decades.

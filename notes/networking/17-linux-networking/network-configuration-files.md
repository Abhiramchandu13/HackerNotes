# Network Configuration Files

> Network configuration files are plain text files in Linux that store persistent network settings for interfaces, IP addresses, DNS servers, and routing tables, ensuring settings are applied automatically on boot.

## Overview
While commands like `ip addr` or `ifconfig` can temporarily configure network settings, those changes are lost upon reboot. To make network configurations persistent, Linux uses various plain text files. These files are read by different networking services (like `NetworkManager`, `systemd-networkd`, or traditional `init` scripts) during system startup to apply the desired settings.

## Why It Matters
For any Linux server or workstation, network configuration files are the source of truth for its identity on the network. Without correctly configured files, a server cannot get an IP, resolve DNS, or connect to the Internet. For security professionals, these files can reveal internal IP ranges, DNS servers, static routes to management networks, and VPN configurations.

## Core Concepts
- **Distribution Specific:** The exact location and format of these files vary significantly between Linux distributions. Ubuntu uses Netplan, CentOS/RHEL use NetworkManager or `network-scripts`, Debian uses `/etc/network/interfaces`.
- **Systemd Integration:** Modern Linux distributions rely heavily on `systemd` to manage services. The `systemd-networkd` service is a common manager for network interfaces.
- **Order of Precedence:** If multiple tools (e.g., `NetworkManager` and manual `ifup`) try to manage the same interface, configuration conflicts can arise.

## How It Works (Conceptual Flow on Boot)
1. Linux boots.
2. The `systemd` init system starts the primary network management service (e.g., `NetworkManager`).
3. The network manager reads its configuration files (e.g., `/etc/NetworkManager/system-connections/`).
4. It then applies the defined settings: bringing up the `eth0` interface, assigning `10.1.1.50` to it, setting `10.1.1.1` as the gateway, and `8.8.8.8` as the DNS server.
5. The system can now communicate over the network.

## Components / Types

### `/etc/resolv.conf`
- **Purpose:** Specifies the DNS servers that the system will use for hostname resolution.
- **Content:** A list of `nameserver` IP addresses.
- **Note:** Often automatically generated by DHCP or NetworkManager. Manual edits might be overwritten.

### `/etc/hosts`
- **Purpose:** A local, static mapping of hostnames to IP addresses. Acts as a simple, local DNS cache.
- **Content:** Lines like `127.0.0.1 localhost` or `10.1.1.50 fileserver.corp.local`.
- **Precedence:** Checked *before* DNS servers. Useful for quick overrides or blocking malicious domains.

### `/etc/network/interfaces` (Debian/Ubuntu Legacy)
- **Purpose:** Defines static IP addresses, DHCP settings, and interface configurations.
- **Content:** Text blocks defining interfaces `eth0`, `wlan0`, etc.
- **Example:**
  ```text
  auto eth0
  iface eth0 inet static
      address 192.168.1.50
      netmask 255.255.255.0
      gateway 192.168.1.1
      dns-nameservers 8.8.8.8 8.8.4.4
  ```

### Netplan (Modern Ubuntu)
- **Purpose:** YAML-based configuration for network interfaces. It generates backend configuration for `NetworkManager` or `systemd-networkd`.
- **Content:** `/etc/netplan/*.yaml` files.
- **Example:**
  ```yaml
  network:
    version: 2
    renderer: networkd
    ethernets:
      enp0s3:
        dhcp4: no
        addresses: [192.168.1.50/24]
        gateway4: 192.168.1.1
        nameservers:
            addresses: [8.8.8.8, 8.8.4.4]
  ```

### NetworkManager (RHEL/CentOS/Fedora)
- **Purpose:** A dynamic network daemon that manages wired, wireless, and VPN connections. It stores configuration in `/etc/NetworkManager/system-connections/`.
- **Content:** `.ini`-like files.

## Practical Examples
- **Static IP Server:** A web server needs a permanent IP. The administrator edits `/etc/netplan/01-netcfg.yaml` to assign a static IP address to the `enp0s3` interface, ensuring it never changes after a reboot.
- **Blocking Malicious Sites (Local):** To quickly block a known phishing site, an admin adds an entry to `/etc/hosts`: `127.0.0.1 malicious-site.com`. The OS will now try to connect to the local machine instead of the real phishing site.

## Security Considerations
- **Information Disclosure:** These files contain IPs, DNS servers, internal hostnames, and sometimes even VPN credentials. They are a treasure trove for attackers who gain access to a Linux system.
- **Malware Persistence:** Attackers often modify network configuration files to ensure their malware automatically connects to a C2 server upon reboot or changes the DNS server to a malicious one.
- **File Permissions:** Critical network configuration files should have strict permissions (`chmod 600`) to prevent unauthorized users from viewing or modifying them.

## Commands / Configuration Examples
### Linux (Applying Netplan Changes)
```bash
# Apply changes made to /etc/netplan/*.yaml files
sudo netplan apply

# Verify the current configuration
ip a
ip route show
```

### Linux (Manual `resolv.conf` Edit)
```bash
# Add Google DNS as the primary resolver
sudo echo "nameserver 8.8.8.8" > /etc/resolv.conf
sudo echo "nameserver 8.8.4.4" >> /etc/resolv.conf
```

## Troubleshooting
- **Network Not Coming Up:** If a Linux server boots but has no network, check the configuration files first. Look for typos, incorrect YAML formatting (for Netplan), or conflicts between different network managers.
- **DNS Resolution Fails:** Verify `/etc/resolv.conf` points to the correct, reachable DNS servers. Check `/etc/hosts` for any accidental overrides.
- **Temporary Changes:** Remember that manual changes made with `ip` or `ifconfig` will be lost after a reboot. Always edit the configuration files for persistent changes.

## Summary
Network configuration files are the memory of a Linux system's network identity. By understanding their varied formats and the daemons that consume them, administrators can ensure persistent connectivity, correctly define host roles, and lay the groundwork for secure communication in any Linux environment.
